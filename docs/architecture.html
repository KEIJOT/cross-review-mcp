How It's Built

The system design explained visually. Every piece, every connection, how data flows from your question to the final answer.

01 — The Big Picture

System Overview

The whole system has three parts: clients (you), the server (the brain), and AI providers (the 5 models). Here's how they connect:

Three columns: who asks (clients), what processes (server), who answers (AI providers)

The server is like a post office. You drop off a letter (your question). The post office makes 5 copies and sends them to 5 different experts at the same time. As each expert replies, the post office collects the answers, compares them, and hands you a summary: "4 out of 5 said the same thing."

02 — The Pieces

System Components

Each file does one job. Here's what each piece is responsible for:

src/index.ts
MCP Server

The front door. Receives requests, registers the 4 tools, decides whether to use stdio or HTTP mode. Starts up the whole system.

src/executor.ts
Review Executor

The brain. Takes your question, sends it to all 5 models at once, and collects their answers. Fires events as each model responds so the dashboard updates live.

src/consensus-algorithm.ts
Consensus Engine

The judge. Looks at all 5 answers, finds what they agree on, scores confidence, and flags interesting disagreements. Uses word-similarity matching to cluster responses.

src/events.ts
Event Bus

The announcement system. Every time something happens (request started, model finished, etc.), it broadcasts to anyone listening — like the dashboard.

src/providers.ts
Provider Layer

The translator. Each AI company has a slightly different way to talk to them. This layer speaks all their languages so the executor doesn't have to.

src/server.ts
HTTP Server

The web server. Runs the dashboard, provides the stats API, streams live events via SSE, and handles remote MCP connections.

src/cache.ts
LRU Cache

The memory. If you ask the same question twice, it returns the saved answer instantly — no API calls, no cost. Saves to disk so it survives restarts.

src/cost-manager.ts
Cost Manager

The accountant. Tracks how much each model costs per question, tallies daily and monthly spend, and can warn you if you're spending too much.

03 — Step by Step

Request Lifecycle

Every question follows this exact path through the system. Here's what happens from the moment you ask to the moment you get your answer:

Your question flows left to right: received → checked in cache → sent to all models at once → parsed → consensus built → answer returned

The whole thing takes 3–5 seconds. The cache check is instant. Sending to 5 models takes 2–8 seconds (but they run at the same time, so it's as fast as the slowest one). Building consensus takes about 100 milliseconds.

04 — Network & Transport

How Clients Connect

Three transport modes let the server work in different situations:

stdio
--mode stdio

Direct pipe to Claude Desktop. Like a private phone line — fastest, simplest.

HTTP
--mode http

Web server on port 6280. Dashboard + API. Anyone on your network can access it.

Both
--mode both

Runs stdio AND HTTP at the same time. Best of both worlds — no conflicts.

HTTP Mode Network Diagram

HTTP mode: the Express server handles the dashboard, API, and MCP endpoint — all on port 6280

05 — Design Decisions

Why It's Built This Way

Every architecture has trade-offs. Here are the key decisions and why they were made:

Parallel execution, not sequential

All 5 models run at the same time. Your total wait time = the slowest model (~3s), not the sum of all 5 (~15s). This is the core insight that makes the tool practical.

Partial success is OK

If one model fails (network error, timeout, bad key), the other 4 still return results. You get the best answer possible instead of "error, try again."

Disagreement is a feature, not a bug

When 4 models agree and 1 disagrees, the different opinion is surfaced as an "alternative perspective" — not hidden. Sometimes the dissenting model is right.

Lazy loading

The system doesn't connect to AI providers when it starts up — only when you make your first request. This prevents startup errors if a key is missing.

Console suppression in stdio mode

When talking to Claude Desktop via stdio, ALL console output is silenced. Otherwise log messages would corrupt the MCP protocol (which uses the same pipe).
