## Feature #1: CACHING LAYER

### What Is It?

A **memory cache** that stores review results so identical reviews don't hit the API twice.

### The Problem It Solves

```
Without caching:
┌─────────────────────────────────────┐
│ User asks: "Review this code"       │ → API call #1: $0.0049
└─────────────────────────────────────┘
         ⏱️ 1 hour later...
┌─────────────────────────────────────┐
│ User asks: "Review this code"       │ → API call #2: $0.0049 (AGAIN!)
│ (exact same code)                   │
└─────────────────────────────────────┘
TOTAL: $0.0098 (wasted 50% on duplicate)
```

With caching:
```
┌─────────────────────────────────────┐
│ User asks: "Review this code"       │ → API call #1: $0.0049
│ Result cached                       │   (stored in memory)
└─────────────────────────────────────┘
         ⏱️ 1 hour later...
┌─────────────────────────────────────┐
│ User asks: "Review this code"       │ → Cache HIT ✅
│ (exact same code)                   │   Return instantly: $0 cost!
│ Instant result from cache           │   10ms instead of 1,250ms
└─────────────────────────────────────┘
TOTAL: $0.0049 (50% savings!)
```

### How It Works (Technical Flow)

```
User submits review request
         ↓
┌────────────────────────────────┐
│ 1. Hash the input content      │
│    hash = SHA256(code)         │
│    hash = "abc123def456"       │
└────────────────────────────────┘
         ↓
┌────────────────────────────────┐
│ 2. Check cache: Does this hash │
│    exist and is it fresh?      │
│    if (cache.has("abc123")     │
│    && !expired)                │
└────────────────────────────────┘
         ↓ YES → Return cached result (INSTANT)
         ↓ NO → Continue to API call
         ↓
┌────────────────────────────────┐
│ 3. Call API (normal flow)      │
│    send request to OpenAI      │
│    send request to Gemini      │
│    wait for responses          │
└────────────────────────────────┘
         ↓
┌────────────────────────────────┐
│ 4. Cache the result            │
│    cache.set("abc123",         │
│      result,                   │
│      ttl: 24 hours)            │
└────────────────────────────────┘
         ↓
Return result to user
```

### How Users Access It

**For developers using the SDK**:

```typescript
import { ReviewExecutor } from 'cross-review-mcp';
import { CacheManager } from 'cross-review-mcp';

// Initialize cache
const cache = new CacheManager({
  ttl: 86400,  // 24 hours
  maxSize: 1000  // max 1000 cached reviews
});

const executor = new ReviewExecutor(config, tracker, cache);

// First request (cache MISS)
const result1 = await executor.execute({
  content: 'Review this code...',
  useCache: true
});
// Result: API call made, took 1,250ms, cost $0.0049

// Second request (identical) (cache HIT)
const result2 = await executor.execute({
  content: 'Review this code...',  // SAME content
  useCache: true
});
// Result: Cached response returned instantly, took 5ms, cost $0!
```

**For Claude Desktop users**:

Add to your config:
```json
{
  "cache": {
    "enabled": true,
    "ttl": 86400,
    "maxSize": 1000
  }
}
```

Then use normally — caching happens **automatically**.

### Configuration Options

```json
{
  "cache": {
    "enabled": true,
    "ttl": 86400,         // seconds (24 hours)
    "strategy": "lru",    // LRU = Least Recently Used
    "maxSize": 1000,      // max entries before eviction
    "persistToDisk": false  // save to SQLite for persistence
  }
}
```

### Real-World Impact

**Scenario**: You review 100 documents, 20 are duplicates

```
Without caching:
  100 API calls × $0.0049 = $0.49 cost, 130s total time

With caching:
  80 API calls × $0.0049 + 20 cache hits = $0.39 cost, 80s total time
  → 20% cost savings, 38% faster
```

**Scenario**: You review the same 10 documents daily for a week

```
Without caching:
  70 API calls × $0.0049 = $0.34/week

With caching:
  10 API calls × $0.0049 = $0.049/week
  → 85% savings per week!
```

## Feature #2: RATE LIMITING

### What Is It?

A **traffic throttle** that prevents you from exceeding API rate limits and getting blocked.

### The Problem It Solves

APIs have quotas:
- OpenAI: 100 requests per minute
- Gemini: 50 requests per minute

Without rate limiting:

```
Second 1:  API calls: 50 → OK
Second 2:  API calls: 50 → OK  
Second 3:  API calls: 60 → ⚠️ WARNING (over quota)
Second 4:  API calls: 40 → ❌ ERROR 429 "Too Many Requests"
                          All subsequent requests FAIL for 60 seconds
```

With rate limiting:

```
Second 1:  API calls: 50 → OK
Second 2:  API calls: 50 → OK
Second 3:  Incoming request: Check quota...
           -> Quota full! 
           -> Wait 0.1 seconds
           -> Try again → OK
```

### How It Works (Technical Flow)

**Token Bucket Algorithm**:

```
┌─────────────────────────────────────────┐
│ RATE LIMITER (OpenAI: 100 req/min)     │
├─────────────────────────────────────────┤
│                                         │
│  Tokens Available: [███████    ] 70/100 │  ← Refills at 100/60s
│                                         │
│  Next Request: ALLOWED (takes 1 token) │
│  Tokens After: [██████     ] 69/100     │
│                                         │
└─────────────────────────────────────────┘

Every 0.6 seconds, 1 new token is added (100 per minute)
```

### Detailed Flow

```
Request comes in for OpenAI
         ↓
┌────────────────────────────────┐
│ Check: Do we have tokens?      │
│ if (tokens > 0)                │
└────────────────────────────────┘
         ↓ YES
┌────────────────────────────────┐
│ Consume 1 token                │
│ tokens = tokens - 1            │
│ Send API request               │
└────────────────────────────────┘
         ↓ Return to user
         
         ↓ NO (tokens = 0)
┌────────────────────────────────┐
│ Wait until token available     │
│ Calculate: nextAvailableTime   │
│ = now + (0.6s per token)       │
│ Sleep until that time          │
└────────────────────────────────┘
         ↓
┌────────────────────────────────┐
│ Tokens refilled? Yes!          │
│ Consume 1 token                │
│ Send API request               │
└────────────────────────────────┘
         ↓ Return to user
```

### How Users Access It

**For developers**:

```typescript
import { RateLimiter } from 'cross-review-mcp';

const limiter = new RateLimiter({
  openai: {
    requestsPerMinute: 100
  },
  gemini: {
    requestsPerMinute: 50
  }
});

const executor = new ReviewExecutor(config, tracker, limiter);

// Use normally — rate limiting is transparent
const result = await executor.execute({
  content: 'Review this...'
});

// Behind the scenes:
// 1. Checks if OpenAI has quota
// 2. If not, waits until it does
// 3. Sends request
// This prevents 429 errors ✅
```

**For Claude Desktop users**:

Add to config:
```json
{
  "rateLimits": {
    "openai": {
      "requestsPerMinute": 100
    },
    "gemini": {
      "requestsPerMinute": 50
    }
  }
}
```

### Real-World Impact

**Scenario**: Batch of 300 reviews submitted

```
Without rate limiting:
  Overwhelming the API → 429 errors → all fail → user frustrated

With rate limiting:
  300 reviews processed at sustainable rate
  OpenAI: 100/min = 3 minutes to complete
  Gemini: 50/min = 6 minutes to complete
  No errors, predictable behavior ✅
```

## Feature #3: BATCH PROCESSING API

### What Is It?

An **async job queue** that lets you submit 100s of reviews, process them in parallel, and track progress.

### The Problem It Solves

Without batch API:

```
User: "I have 100 reviews to do"
Code: for (let i = 0; i < 100; i++) {
        const result = await executor.execute(reviews[i]);
      }

Timeline:
Review 1:  Takes 1.3s  (total: 1.3s)
Review 2:  Takes 1.3s  (total: 2.6s)
Review 3:  Takes 1.3s  (total: 3.9s)
...
Review 100: Takes 1.3s (total: 130 seconds!) 😫
```

With batch API:

```
User: "Process 100 reviews"
Code: const batchId = await executor.executeBatch(reviews);

Timeline:
All reviews start in parallel
Review 1-10: Running simultaneously
  ~ Takes 1.3s for all to complete
Review 11-20: Running simultaneously
  ~ Takes 1.3s for all to complete
...
Total: ~13 seconds (10x faster!) 🚀
```

### How It Works (Technical Flow)

```
User submits batch of 100 reviews
         ↓
┌────────────────────────────────────────┐
│ 1. CREATE BATCH JOB                    │
│    batchId = "batch-abc123"            │
│    status = "QUEUED"                   │
│    Store in database:                  │
│    {                                   │
│      id: "batch-abc123",               │
│      totalReviews: 100,                │
│      completed: 0,                     │
│      results: [],                      │
│      createdAt: now                    │
│    }                                   │
└────────────────────────────────────────┘
         ↓ Return batchId to user IMMEDIATELY
         ↓
┌────────────────────────────────────────┐
│ 2. QUEUE REVIEWS                       │
│    Add all 100 reviews to job queue    │
│    Each review is independent task     │
└────────────────────────────────────────┘
         ↓
┌────────────────────────────────────────┐
│ 3. PROCESS IN PARALLEL                 │
│    Worker 1: Process review 1-10       │
│    Worker 2: Process review 11-20      │
│    Worker 3: Process review 21-30      │
│    Worker 4: Process review 31-40      │
│    (respecting rate limits)            │
│                                        │
│    As each completes:                  │
│    - Store result in database          │
│    - Increment batch.completed         │
│    - Trigger webhook (if configured)   │
└────────────────────────────────────────┘
         ↓
User polls for status (or gets webhook notification)
         ↓
┌────────────────────────────────────────┐
│ 4. RETRIEVE RESULTS                    │
│    const status =                      │
│      await executor.getBatchStatus(    │
│        "batch-abc123"                  │
│      );                                │
│    {                                   │
│      status: "PROCESSING",             │
│      completed: 45,                    │
│      total: 100,                       │
│      progress: 45%,                    │
│      results: [...]                    │
│    }                                   │
└────────────────────────────────────────┘
         ↓
When complete → User downloads results
```

### How Users Access It

**For developers**:

```typescript
import { ReviewExecutor } from 'cross-review-mcp';

const executor = new ReviewExecutor(config, tracker);

// Step 1: SUBMIT batch
const reviews = [
  { content: 'Code 1...' },
  { content: 'Code 2...' },
  { content: 'Code 3...' },
  // ... 97 more
];

const batchId = await executor.executeBatch(reviews, {
  name: 'MyBatch',
  notifyWebhook: 'https://myserver.com/webhook'
});

console.log(`Batch submitted: ${batchId}`);
// Output: Batch submitted: batch-abc123def456
// RETURNS IMMEDIATELY (async job running in background)

// Step 2: POLL for status
const pollStatus = async (batchId) => {
  const status = await executor.getBatchStatus(batchId);
  
  if (status.status === 'PROCESSING') {
    console.log(`Progress: ${status.completed}/${status.total}`);
    // Output: Progress: 45/100
    
    setTimeout(() => pollStatus(batchId), 5000); // Check every 5s
  } else if (status.status === 'COMPLETE') {
    console.log(`Batch done! Results:`, status.results);
    // Process results
  }
};

pollStatus(batchId);

// Step 3: DOWNLOAD results (when complete)
const results = await executor.getBatchResults(batchId);
results.forEach(r => {
  console.log(`Review: ${r.testName} - Cost: $${r.cost}`);
});
```

**For Claude Desktop users**:

```
Batch submission:
  1. Create batch.json with array of reviews
  2. Call: /batch/submit (reads batch.json)
  3. Get: batchId
  
Monitoring:
  4. Call: /batch/status/{batchId}
  5. Returns: { completed: 50, total: 100, progress: 50% }
  
Retrieval:
  6. Call: /batch/results/{batchId}
  7. Get: All results as JSON
```

### Configuration

```json
{
  "batch": {
    "enabled": true,
    "parallelWorkers": 4,     // Process 4 reviews simultaneously
    "workerTimeoutMs": 120000, // 2 minute timeout per review
    "persistDatabase": true,   // Save to SQLite
    "maxBatchSize": 10000      // Max 10k reviews per batch
  }
}
```

### Real-World Comparison

**Sequential vs Batch Processing**:

```
SEQUENTIAL (current):
  100 reviews × 1.3s each = 130 seconds (2 min 10 sec)
  Cost: 100 API calls = $0.49
  
BATCH (with 4 workers):
  25 batches of 4 reviews, each takes 1.3s = 32.5 seconds (6x faster!)
  Cost: Still 100 API calls = $0.49
  Throughput: 3 reviews/second vs 0.77 reviews/second
  
Network:
  Sequential: 100 separate network requests
  Batch: 25 requests (parallelized)
```

## Questions to Validate Understanding

1. **Caching**: Why doesn't caching increase cost?
   - A: Because cached results don't hit the API (no API call = no cost)

2. **Rate Limiting**: What happens if you exceed rate limits?
   - A: Without it: API returns 429 error, request fails
   - A: With it: System waits until quota resets, then retries

3. **Batch API**: Can you process 1000 reviews instead of 100?
   - A: Yes, same system. Just takes longer (6+ minutes vs 30s)

4. **All Together**: If you batch 100 reviews daily for 7 days (with 30% duplicates):
   - Caching saves: 30% × 7 days = ~$1.03
   - Batch saves time: 130s → 30s per batch = 7 × 100s saved
   - Rate limiting prevents: Errors and quota violations
